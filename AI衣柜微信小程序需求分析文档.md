# AI 衣柜微信小程序需求分析文档

版本：V1.1  
日期：2026-05-11  
产品形态：微信小程序优先，后续可扩展 App  
AI 接口：`chatgptimage2`，按 OpenAI `gpt-image-2` 图像生成/编辑能力设计接口适配层

---

## 1. 项目概述

### 1.1 项目名称

暂定名称：搭搭衣橱

### 1.2 产品定位

一款低成本、轻量化的 AI 衣柜微信小程序，帮助用户把自己的衣服、商品图、网图或截图上传到数字衣柜，生成穿搭组合，并使用 `chatgptimage2` 接口生成穿搭预览图。

产品不做复杂重资产的电商平台，也不在第一版做完整 App。第一阶段聚焦三个高频动作：

1. 录入衣服
2. 推荐搭配
3. 生成试穿预览图并分享

### 1.3 核心价值

用户每天最关心的问题不是“我有多少衣服”，而是“我今天穿什么”。本产品通过数字衣橱和 AI 穿搭图生成，把用户已有衣服重新利用起来，降低选择成本，增加分享和复购可能。

### 1.4 建设原则

- 小程序先行，降低获客、开发和分发成本。
- 先做搭配决策，再做高质量试穿，避免 AI 成本失控。
- 图片生成接口只在关键场景调用，普通搭配推荐优先使用本地规则和缓存。
- 所有 AI 调用走后端，不在小程序端暴露 API Key。
- 用户上传全身照后，该照片对应的人物自动成为“我的模特”；未上传全身照时，系统默认使用普通模特。
- 页面设计参考 Google 系产品的简洁感：白底、图片优先、轻量卡片、少文字、低干扰。
- 明确提示 AI 试穿为视觉参考，不承诺真实尺码、面料垂感和上身效果完全一致。

## 2. 目标用户与使用场景

### 2.1 目标用户

| 用户类型 | 典型特征 | 核心需求 |
| --- | --- | --- |
| 普通女性用户 | 衣服多，但每天不知道怎么搭 | 今日穿搭、场合搭配、朋友投票 |
| 通勤用户 | 穿衣重复、希望省时间 | 一键生成通勤搭配 |
| 学生用户 | 预算有限、喜欢分享 | 用已有衣服搭出新感觉 |
| 穿搭博主/店主 | 需要快速做搭配图 | 生成穿搭示意图、分享图 |
| 小服装店 | 缺少模特图预算 | 用商品图生成搭配参考图 |

### 2.2 高频场景

1. 早上出门前：用户打开小程序，查看“今日推荐穿搭”。
2. 买衣服前：用户上传商品图，查看能否和现有衣服搭配。
3. 出门聚会前：用户生成 2 到 3 套穿搭图，发给朋友投票。
4. 旅行前：用户选择目的地和天数，小程序生成轻量行李搭配清单。
5. 店主上新：上传商品图，快速生成搭配参考图用于朋友圈或私域。

| 搭配推荐 | 基于规则生成上衣+下装+鞋包组合 | P0 |
| 我的模特 | 上传全身照后保存为用户专属模特 | P0 |
| AI 试穿图 | 用专属模特或默认普通模特生成穿搭预览 | P0 |

- 不做完整 App。
- 不做自动扫描用户全部相册。
- 不做真实尺码推荐。
- 不做复杂 3D 试衣。

- 自动识别衣物分类和颜色。
- 天气穿搭推荐。
- 旅行胶囊衣橱。
- 好友投票。
- 商品推荐和 CPS 分佣。
- 店主版批量生成。
- App 端相册批量导入。

## 4. 核心业务流程

### 4.1 新用户流程

1. 用户进入小程序。
2. 微信授权登录。
3. 展示隐私说明和图片使用说明。
4. 引导上传 3 到 5 件衣服，可上传自己拍的衣服、商品图、网图截图。
5. 用户给衣服选择基础标签。
6. 系统生成第一套穿搭推荐。
7. 用户选择上传全身照，上传后自动保存为“我的模特”。
8. 如果用户不上传全身照，系统使用默认普通模特。
9. 用户选择是否生成 AI 试穿图。
10. 生成完成后保存到“我的穿搭”。

### 4.2 AI 试穿流程

1. 用户进入 AI 试穿页。
2. 系统检查用户是否已有“我的模特”。
3. 如果已有专属模特，默认选中该模特。
4. 如果没有专属模特，默认选中普通模特。
5. 用户选择衣柜中的上衣、下装、鞋包等单品。
6. 前端提交生成任务。
7. 后端检查用户额度、图片合规、参数完整性。
8. 后端创建生成任务，进入队列。
9. 后端调用 `chatgptimage2` 图像编辑/生成接口。
10. 生成成功后保存图片到云存储。
11. 前端轮询或订阅任务状态。
12. 用户查看、保存、分享生成结果。

### 4.3 失败降级流程

| 失败类型 | 处理方式 |
| --- | --- |
| 图片不清晰 | 提示重新上传，并不扣次数 |
| 图片违规 | 阻断生成，提示更换图片 |
| API 超时 | 自动重试 1 次，仍失败则退还次数 |
| API 额度不足 | 提示稍后再试或切换备用服务 |
| 生成效果差 | 允许用户点“不满意”，同任务免费重试 1 次 |
| 网络中断 | 任务继续在后端执行，用户回到生成记录查看 |

## 5. 功能需求

## 5.1 首页

### 功能说明

首页展示今日穿搭推荐、AI 试穿入口、衣柜概览和最近生成结果。

### 页面元素

- 今日推荐穿搭
- AI 试穿按钮
- 上传衣服按钮
- 我的衣柜分类入口
- 最近生成图片
- 免费生成次数提示

### 验收标准

- 用户 3 秒内可以找到“上传衣服”和“AI 试穿”入口。
- 没有衣服时展示空状态和上传引导。
- 已有衣服时优先展示推荐搭配。

## 5.2 衣物上传

### 功能说明

用户可以通过拍照、相册、网图截图、商品图等方式上传衣物图片。第一版建议主流程只做“本地图片上传”，也就是用户把网图或商品图保存到手机后再上传；“粘贴图片链接自动抓取”可作为二期能力，避免第一版被跨域、盗链、版权和图片解析问题拖慢。

### 上传来源

| 来源 | 第一版处理方式 | 说明 |
| --- | --- | --- |
| 自己拍摄 | 直接拍照上传 | 适合真实衣物 |
| 手机相册 | 从相册选择 | 适合已保存图片 |
| 网图截图 | 作为相册图片上传 | 适合灵感图、商品图 |
| 电商商品图 | 保存到相册后上传 | 仅用于个人搭配参考 |
| 图片 URL | 二期支持 | 后端下载、审核、存储 |

### 字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| 图片 | image | 是 | 衣服实拍图、商品图、网图截图 |
| 来源 | enum | 是 | 拍照、相册、网图、商品图 |
| 分类 | enum | 是 | 上衣、下装、裙装、鞋、包、配饰 |
| 主色 | enum | 是 | 黑、白、灰、蓝、红、绿、棕等 |
| 季节 | enum | 否 | 春、夏、秋、冬、四季 |
| 场合 | enum | 否 | 通勤、休闲、约会、运动、正式 |
| 原始链接 | url | 否 | 二期支持，记录商品图或网图来源 |
| 备注 | text | 否 | 用户自定义描述 |

### 验收标准

- 上传成功后 2 秒内展示在衣柜列表。
- 图片上传失败时可重新上传。
- 分类和主色必须填写，防止搭配推荐失效。
- 用户上传网图或商品图时，页面提示“仅用于个人穿搭参考，商用需确认图片权利”。

## 5.3 我的衣柜

### 功能说明

用户查看、筛选、编辑、删除自己的衣服。

### 筛选条件

- 分类
- 颜色
- 季节
- 场合
- 最近使用

### 验收标准

- 支持按分类快速切换。
- 删除衣物前二次确认。
- 删除衣物不影响历史生成图片，但历史记录中标记“原衣物已删除”。

## 5.4 搭配推荐

### 功能说明

系统基于衣物标签生成穿搭组合。第一版不依赖 AI 大模型做所有推荐，以降低成本。

### 推荐规则

- 上衣 + 下装 + 鞋为基础组合。
- 连衣裙可直接搭配鞋和包。
- 全身颜色不超过 3 个主色。
- 黑、白、灰、牛仔蓝作为高兼容色。
- 场合优先级高于颜色。
- 季节不匹配时不推荐。

### 验收标准

- 用户衣柜中至少有 1 件上衣、1 件下装、1 双鞋时，可生成基础搭配。
- 用户衣物不足时，提示缺少的分类。
- 每次推荐至少返回 3 套组合，衣物不足时返回可用组合。

## 5.5 我的模特

### 功能说明

用户上传一张清晰全身照后，系统自动将该人物保存为“我的模特”。后续 AI 试穿默认使用这个专属模特生成效果图。用户没有上传全身照时，系统使用普通默认模特。

### 模特类型

| 类型 | 说明 | 使用场景 |
| --- | --- | --- |
| 我的模特 | 用户上传的本人或授权人物全身照 | 查看自己上身效果 |
| 默认模特 | 系统内置普通模特 | 新用户、未上传全身照、只想看搭配效果 |

### 上传要求

- 照片应为清晰全身照。
- 人物应正面或轻微侧身站立。
- 光线正常，身体无遮挡。
- 不建议上传多人合照。
- 不允许上传未授权他人照片。
- 上传成功并通过审核后，自动设为当前默认模特。

### 页面元素

- 当前模特预览
- 上传全身照按钮
- 使用默认模特按钮
- 更换模特按钮
- 删除我的模特按钮
- 隐私说明入口

### 验收标准

- 用户上传全身照并审核通过后，AI 试穿页默认选中“我的模特”。
- 用户未上传全身照时，AI 试穿页默认选中普通模特。
- 用户可以删除我的模特，删除后自动回到默认模特。
- 我的模特图片不能被其他用户访问。
- 分享结果页只展示生成图，不暴露原始全身照。

## 5.6 AI 试穿图生成

### 功能说明

用户选择“我的模特”或普通默认模特，再选择衣物组合后，系统调用 `chatgptimage2` 生成穿搭预览图。

### 生成模式

| 模式 | 用途 | 成本策略 |
| --- | --- | --- |
| 快速预览 | 免费用户、低清预览 | 低质量/较小尺寸，限制次数 |
| 高清生成 | 会员或付费单次 | 高质量，用户确认后调用 |
| 商品搭配图 | 店主或达人 | 可批量，但需付费套餐 |

### 模特使用规则

| 条件 | 系统行为 |
| --- | --- |
| 用户已有我的模特 | 默认使用我的模特生成 |
| 用户没有我的模特 | 默认使用普通模特生成 |
| 用户主动切换默认模特 | 本次任务使用默认模特 |
| 用户删除我的模特 | 后续任务自动使用默认模特 |
| 用户重新上传全身照 | 新照片审核通过后成为新的我的模特 |

### Prompt 生成要求

后端根据结构化参数拼接 Prompt，避免前端直接传任意文本。

基础 Prompt 模板：

```text
Create a realistic outfit preview based on the provided person photo and clothing item images.
Keep the person's face, body pose, body shape, skin tone, and background as consistent as possible.
Apply the selected outfit naturally with realistic fabric folds, lighting, and proportions.
Do not change identity. Do not create nudity. Do not add unrelated accessories.
Output a clean full-body fashion preview suitable for a wardrobe app.
If no personal model photo is provided, use a natural ordinary fashion model with a simple neutral background.
```

中文业务说明：

- 保持人物身份、姿势、脸部和背景尽量一致。
- 替换或叠加用户选择的衣物。
- 用户有“我的模特”时，尽量保持该人物特征。
- 用户没有“我的模特”时，使用普通自然模特，不生成夸张网红脸或过度修饰。
- 不生成裸露、低俗、未成年人敏感内容。
- 不添加用户未选择的夸张配饰。
- 输出适合小程序展示的完整穿搭预览图。

### 验收标准

- 任务提交后立即返回任务 ID。
- 生成中页面展示预计等待时间。
- 成功后展示结果图、保存和分享按钮。
- 失败时展示失败原因，并按规则退还次数。
- 每次调用必须记录请求参数、状态、耗时、错误码和成本估算。
- 生成结果必须标记本次使用的是“我的模特”还是“默认模特”。

## 5.7 生成记录

### 功能说明

保存用户历史生成的 AI 穿搭图。

### 字段

| 字段 | 说明 |
| --- | --- |
| 生成图片 URL | 云存储地址 |
| 使用的模特 ID | 我的模特或默认模特 |
| 使用的衣物 ID | 关联衣物 |
| 模特照片 ID | 关联我的模特照片，默认模特可为空 |
| 生成模式 | 快速预览/高清生成 |
| 任务状态 | 排队中、生成中、成功、失败 |
| 失败原因 | API 超时、图片违规、参数错误等 |
| 创建时间 | 生成时间 |

### 验收标准

- 用户可查看最近 30 天生成记录。
- 用户可删除单条记录。
- 删除后云存储图片进入异步清理队列。

## 5.8 分享功能

### 功能说明

用户可以将 AI 穿搭图生成分享卡片，发送给微信好友或群。

### 分享卡片内容

- 穿搭图
- 小程序码
- 简短标题，如“帮我看看今天穿这套可以吗？”
- 可选投票入口，二期实现

### 验收标准

- 分享卡片打开后能进入对应结果页。
- 非本人访问时只能查看分享图，不能查看用户衣柜隐私数据。

## 5.9 付费与额度

### 免费策略

- 新用户赠送 3 次快速预览。
- 每日签到赠送 1 次快速预览，可设置上限。
- 高清生成需要消耗更多次数或付费。

### 付费策略

| 套餐 | 建议权益 |
| --- | --- |
| 单次生成包 | 10 次高清生成 |
| 月卡 | 每日固定快速预览 + 高清额度 |
| 店主包 | 批量生成、去水印、商用提示 |

### 成本控制

- 免费用户默认低质量或小尺寸生成。
- 同一套衣服和同一用户照片重复生成时优先读缓存。
- 每个用户设置每日上限。
- API 异常时暂停新任务，防止连续失败扣费。
- 后台配置每日总预算上限，超过后暂停免费生成。

## 6. 非功能需求

### 6.1 性能

- 小程序首页首屏加载时间不超过 2 秒。
- 衣柜列表分页加载，每页 20 件。
- AI 生成任务异步执行，不阻塞前端。
- 生成任务状态查询间隔建议 2 到 3 秒。

### 6.2 安全

- `chatgptimage2` API Key 只能保存在服务端环境变量中。
- 小程序端不得直接请求第三方 AI API。
- 用户上传图片必须鉴权访问。
- 图片 URL 使用临时签名或私有读策略。
- 管理后台必须限制 IP 或启用二次验证。

### 6.3 隐私

- 用户上传人像前必须展示用途说明。
- 用户可以删除个人照片、衣物图片和生成结果。
- 用户上传全身照后，应明确提示“该照片将作为我的模特用于 AI 试穿”。
- 用户注销后，个人数据进入异步删除流程。
- 不默认读取用户完整相册。
- 不将用户图片用于模型训练，除非获得单独明确授权。
- 默认模特不需要用户授权，但需要在页面上标注“默认模特”。

### 6.4 内容安全

- 上传前或上传后进行图片安全审核。
- 对人像、低俗、暴力、未成年人敏感内容进行拦截。
- Prompt 模板中禁止生成裸露、色情、违法内容。
- 生成结果也需要安全审核后再展示。
- 网图和商品图允许作为个人搭配参考素材上传，但商用导出或店主套餐需提示用户确认图片权利。

### 6.5 稳定性

- AI 生成任务必须有状态机。
- 每个任务最多自动重试 1 次。
- 接口超时后不重复扣用户额度。
- 第三方接口异常时返回统一错误码。
- 所有生成失败都要可追踪。

## 7. 技术方案

### 7.1 推荐技术架构

| 层级 | 推荐方案 | 原因 |

### 7.3 `chatgptimage2` 接口设计

根据 OpenAI 官方文档，`gpt-image-2` 属于图像生成/编辑模型，支持文本输入和图片输入/输出，适合本项目中的“穿搭预览图生成”和“图片编辑式试穿”能力。实际开发中不要让业务代码直接依赖第三方参数，应增加统一适配层。

#### 后端内部接口

```http
POST /api/ai/outfit-render
```

请求参数：

```json
{
  "modelType": "personal_model",
  "modelPhotoId": "model_001",
  "clothingItemIds": ["cloth_top_001", "cloth_bottom_002", "shoes_003"],
  "mode": "quick",
  "scene": "commute",
  "style": "clean_realistic",
  "shareable": true
}
```

响应参数：

```json
{
  "taskId": "task_202605110001",
  "status": "queued",
  "estimatedSeconds": 30
}
```

#### 任务查询接口

```http
GET /api/ai/tasks/{taskId}
```

响应参数：

```json
{
  "taskId": "task_202605110001",
  "status": "success",
  "imageUrl": "https://cdn.example.com/result.png",
  "errorCode": null,
  "errorMessage": null
}
```

#### 适配器职责

- 拼接安全 Prompt。
- 读取用户模特图、默认模特图和衣物图。
- 调用 `chatgptimage2` 的图片生成或图片编辑接口。
- 处理超时、限流、失败重试。
- 统一返回图片结果。
- 记录消耗、耗时和错误。

### 7.4 为什么要做适配层

如果未来 `chatgptimage2` 改名、限流、涨价，或者需要切换到其他图像生成接口，只需要修改适配层，不影响小程序页面、订单系统和用户数据。

### 9.1 页面设计风格

整体参考 Google 系产品的简洁、干净、图片优先体验，但不直接照搬具体界面。

### 视觉原则

- 白色或极浅灰背景。
- 首页首屏只保留 2 个主入口：“上传衣服”和“AI 试穿”。
- 图片占主要视觉空间，文字尽量短。

- 衣柜用瀑布流或两列网格展示，不做复杂装饰。
- 卡片圆角控制在 8px 左右，避免过度可爱化。
- 底部导航建议 4 个：首页、衣柜、试穿、我的。
- 免费次数、生成状态、隐私提示以轻提示展示，不抢主内容。

### 简洁约束

- 首页不放营销长文案。

- 不使用大面积彩色渐变。

- 同一页面主按钮不超过 2 个。
- 生成页只展示必要状态：排队中、生成中、完成、失败。

## 18. 防错清单

### 18.1 AI 任务状态机

AI 生成必须使用明确状态，避免重复扣费、重复生成和前端等待异常。

| 状态 | 说明 | 可进入的下一个状态 |
| --- | --- | --- |
| created | 任务已创建，尚未扣额度 | queued/canceled |
| queued | 已扣额度，等待执行 | running/failed |
| running | 正在调用 AI 接口 | success/failed |
| success | 生成成功 | deleted |
| failed | 生成失败 | refunded/retry |
| refunded | 已退还额度 | closed |
| deleted | 用户删除结果 | closed |

### 18.2 标准错误码

| 错误码 | 触发场景 | 用户提示 | 是否退还额度 |
| --- | --- | --- | --- |
| AUTH_REQUIRED | 用户未登录 | 请先登录后再使用 | 不涉及 |
| INVALID_IMAGE | 图片格式、大小或清晰度不合格 | 图片不清晰，请重新上传 | 是 |
| AUDIT_REJECTED | 图片未通过内容安全 | 图片不符合使用规范，请更换 | 是 |
| CREDIT_NOT_ENOUGH | 用户额度不足 | 生成次数不足 | 否 |
| TASK_NOT_FOUND | 查询不存在的任务 | 任务不存在或已删除 | 不涉及 |
| AI_TIMEOUT | AI 接口超时 | 生成超时，次数已退还 | 是 |
| AI_RATE_LIMIT | AI 接口限流 | 当前生成排队较多，请稍后再试 | 是 |
| AI_FAILED | AI 接口返回失败 | 生成失败，次数已退还 | 是 |
| STORAGE_FAILED | 结果图保存失败 | 保存失败，请重新生成 | 是 |
| DUPLICATE_SUBMIT | 用户重复点击提交 | 任务已提交，请勿重复点击 | 否 |

### 18.3 核心测试用例

| 用例 | 操作 | 预期结果 |
| --- | --- | --- |
| 新用户无衣物 | 进入首页 | 展示上传引导，不报错 |
| 衣物不足 | 只有上衣，无下装 | 提示缺少下装或鞋子 |
| 上传网图截图 | 从相册上传商品图或网图 | 保存到衣柜，并标记来源 |
| 未上传全身照 | 点击 AI 试穿 | 默认使用普通模特 |
| 上传全身照 | 上传清晰单人全身照 | 审核通过后成为我的模特 |
| 删除我的模特 | 删除当前模特 | 后续试穿自动回到默认模特 |
| 图片过大 | 上传超大图片 | 压缩或提示重新上传 |
| 内容违规 | 上传违规人像或图片 | 拦截，不进入 AI 生成 |
| 额度不足 | 点击生成 | 提示额度不足，不创建任务 |
| 重复点击 | 连续点击生成按钮 | 只创建一个任务 |
| API 超时 | 模拟 `chatgptimage2` 超时 | 自动重试一次，失败后退还额度 |
| 生成成功 | 正常提交任务 | 保存结果图和生成记录 |
| 删除衣物 | 删除已生成过的衣物 | 历史图保留，衣物显示已删除 |
| 删除账号 | 用户注销 | 个人图片、衣物、生成记录进入删除流程 |
| 分享访问 | 好友打开分享页 | 只能查看分享图，不能访问衣柜 |

### 18.4 上线前检查

- API Key 已放在服务端环境变量。
- 小程序端没有第三方 AI 密钥。
- AI 任务有去重锁，防止重复提交。
- 扣额度和创建任务必须在同一个事务或补偿流程内完成。
- 失败退款必须有流水记录。
- 图片上传和生成结果都经过内容安全。
- 用户未上传全身照时，AI 生成必须能自动使用默认模特。
- 用户删除我的模特后，不得继续使用已删除全身照生成。
- 免费生成有每日总预算上限。
- 管理后台可以一键暂停 AI 生成。
- 用户协议、隐私政策、图片用途说明已配置。
- 所有外部链接、图片域名、接口域名已加入微信小程序合法域名。

---

## 19. 结论

本方案推荐以“微信小程序 + 云开发 + `chatgptimage2` 图像生成适配层”的方式启动。第一版只做衣物录入、规则搭配、AI 试穿图和分享闭环，尽量减少复杂电商、社区和自动识别功能。

这样可以用较小预算在 4 到 6 周内完成 MVP，上线后通过生成次数、分享率、复用率和付费率判断是否继续投入 App、自动识别和店主版能力。
